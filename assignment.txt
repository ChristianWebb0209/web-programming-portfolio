Small Portfolio and Project 
Due Monday by 11:59pm Points 100 Submitting a text entry box or a website url Available May 5 at 12am - May 20 at 11:59pm
Overview
You will build a small personal web site that functions as both a public-facing introduction to who you are and a demonstration of what you can build with the modern web platform. The site should serve as a credible professional landing place — somewhere a recruiter, classmate, or future collaborator can quickly understand who you are, see your résumé, browse a few projects, and reach you. It must also include at least one piece of evidence that you can think and code like a web application developer, not only a page author.

Learning Goals
You will demonstrate fluency with semantic HTML, modern CSS, and vanilla JavaScript; the ability to ship and deploy a real site through a real version control and hosting workflow; and an understanding of progressive enhancement as a design philosophy rather than a checkbox.

Required Content
At minimum the site must include a home or about page introducing you, a résumé page reachable as a route (not solely a downloadable PDF), a projects page listing at least the demonstration component described below, and a working contact mechanism. A blog or writing section is encouraged but not required. Pages must be linked through clear navigation and read as a coherent site rather than a folder of unrelated files.

Markup, Styling, and Behavior
Every page must validate as semantic HTML5. Use elements for what they mean: <article>, <section>, <nav>, <header>, <footer>, <main>, headings in proper order, lists where lists belong, and forms with labels. CSS must be hand-authored using modern features — flexbox and grid for layout, custom properties for tokens and theming, and relative units (rem, em, %, ch, vw/vh, clamp()) for sizing — and must not depend on a CSS framework such as Bootstrap or Tailwind. JavaScript, where used, must be vanilla and applied in a progressive enhancement style: the page must be useful and navigable with scripting disabled, with JavaScript layering on enhancements rather than gating core content.

Tooling and Deployment
You may write the site by hand or use a static site generator. Astro and 11ty are the supported choices; both are lightweight enough to remain platform-aligned. If you adopt an SSG, prefer plain HTML, CSS, and JS over framework components. The site must live in a public GitHub repository with a sensible commit history (not a single dump commit) and a README explaining the project and how to build it. Deployment is required: Cloudflare Pages is preferred, GitHub Pages is acceptable. The deployed URL must be linked from the README.

Contact Form
The contact form must actually work. It must submit somewhere and result in the message reaching you — through a form-handling service, a Cloudflare Worker, a Pages Function, a mailto: fallback, or any other approach you can defend. Validation should use native HTML5 attributes (required, type="email", pattern) before any JavaScript intervenes. The form must remain accessible and submittable without JavaScript.

Web Application Demonstration
The site must include at least one component that demonstrates application-layer thinking. The minimum bar is a small interactive widget that fetches data from a real REST endpoint and renders it: a weather card for a chosen city, a live sports score, currency conversion, a GitHub stats widget, a transit arrivals board, an air quality readout — anything that involves an HTTP request, JSON parsing, error handling, and a sensible loading state. Build this as a small custom element or module of vanilla JavaScript. Frameworks (React, Vue, Svelte, Angular, and so on) are not permitted anywhere in the project.

Extra Credit
More ambitious work is rewarded. Examples include a small CRUD application backed by a Worker and KV or D1, a multi-step interactive experience using View Transitions, a coordinated set of Web Components used across multiple pages, an offline-capable PWA with a service worker, or any nontrivial application that demonstrates depth. As with the required demonstration, all extra-credit work must be built on standard web technology without a framework.

Accessibility and Aesthetics
The site must be usable by keyboard, screen reader, and at common zoom levels. Color contrast must meet WCAG AA. Aesthetics should be consistent across pages — typography, spacing, and color decisions should look intentional rather than accidental — but visual polish is not the primary grading criterion. Correct, valid, semantic markup and clean separation of concerns are weighted more heavily than how striking the design looks.

Deliverables
Submit the URL of your GitHub repository and the URL of the deployed site. The README must document the stack you chose, how to run the site locally, and a brief note on the application demonstration explaining what it does and which API it consumes.


AI Use
AI use is likely so I am simply not going to police it.  However, I will point out that if we assume you can use it any mistakes made with it are yours.  Use Claude Code, Chat GPT, or whatever makes sense and use it to help, but be careful that it doesn't add dependencies, break code, or add complexity that causes you to drop points even if your result looks good it might not meet the rubric. 

Grading
HTML correctness and semantics (25%), CSS quality and modernity (20%), JavaScript quality and progressive enhancement (15%), the application demonstration (20%), deployment and repository hygiene (10%), accessibility (10%). Extra credit can raise the final grade above 100% to a cap announced separately.
